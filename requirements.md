# SSH Job Management App Requirements

## Overview

複数のSSH接続先で実行される学習ジョブやシミュレーションジョブを、一元的に監視・管理するアプリケーションを開発する。

主な目的は以下の通り。

- 複数ホスト上のジョブ状況をまとめて把握できること
- 実行中プロセスの開始・停止・状態確認ができること
- 学習やsimulationの進捗、標準出力・標準エラー、異常終了時のエラー内容を確認できること
- ジョブ完了や異常終了を通知できること

## Target Users

- SSHで複数の計算サーバやGPUサーバを使い分ける開発者
- 学習ジョブやsimulationを長時間実行する研究者・エンジニア
- 複数ホストでプロセス管理をまとめて行いたいユーザー

## Assumed Use Cases

- 複数のSSH先に接続し、それぞれのジョブ一覧をまとめて閲覧する
- 特定ホスト上で動作中の学習ジョブの進捗を確認する
- simulationや学習ジョブが異常終了した際に、エラー文を確認する
- 実行完了時または失敗時に通知を受け取る
- 不要になったジョブを停止する

## Functional Requirements

### 1. SSH Host Management

- 複数のSSH接続先を登録できる
- 各接続先に対して以下の情報を保持できる
  - ホスト名
  - 表示名
  - ポート
  - ユーザー名
  - 認証方式
  - 利用メモ
- 接続先ごとに疎通確認ができる
- 接続先の追加・編集・削除ができる

### 2. Job / Process Management

- 各SSH先で実行中のプロセス一覧を取得できる
- 管理対象ジョブを識別できる
- ジョブに対して以下の操作ができる
  - 開始
  - 停止
  - 再起動
  - 状態確認
- ジョブごとに以下の基本情報を表示できる
  - ジョブ名
  - 実行ホスト
  - プロセスID
  - 開始時刻
  - 実行時間
  - ステータス

### 3. Log / Progress Monitoring

- 実行中ジョブの標準出力ログを確認できる
- 実行中ジョブの標準エラーログを確認できる
- ジョブごとの最新ログを追跡表示できる
- 学習やsimulationの進捗を確認できる
- 進捗の表示方法として以下を検討する
  - ログから進捗情報を抽出して表示
  - ユーザー定義の進捗パターンを設定
  - 進捗率、step数、epoch数、残り時間などの表示

### 4. Failure Handling

- ジョブ異常終了を検知できる
- 異常終了時にエラーログを確認できる
- 直近の標準エラー出力を保存・表示できる
- 異常終了したジョブを一覧で確認できる

### 5. Notifications

- 以下のイベントで通知できる
  - ジョブ開始
  - ジョブ完了
  - ジョブ失敗
  - 長時間無出力
- 通知先は今後の候補として以下を想定する
  - アプリ内通知
  - メール
  - Slack
  - Discord
  - Webhook

### 6. Search / Filter

- ホスト名、ジョブ名、ステータスで絞り込みできる
- 実行中、完了、失敗などでフィルタできる
- ログ内容で検索できる

## Non-Functional Requirements

### 1. Usability

- 複数ホスト・複数ジョブの状態を一覧で把握しやすいUIであること
- 重要な異常状態が視覚的に分かりやすいこと
- 長時間監視でも操作負荷が低いこと

### 2. Reliability

- SSH切断時に再接続またはエラー通知できること
- 一時的な接続失敗時に状態不整合を起こしにくいこと
- ログ取得やジョブ状態取得が部分失敗しても全体監視が継続できること

### 3. Security

- SSH認証情報を安全に扱うこと
- 秘密鍵やトークンを平文で不用意に保存しないこと
- 接続先ごとのアクセス権限を適切に管理できる設計であること

### 4. Performance

- 接続先が複数あっても一覧更新が極端に遅くならないこと
- ログ表示や進捗確認が実用的な速度で行えること

## MVP Scope

まずは以下をMVPとする。

- SSH接続先を複数登録できる
- 接続先ごとにジョブ一覧を取得できる
- 実行中ジョブの標準出力・標準エラーを確認できる
- ジョブの進行状況を最低限ログベースで確認できる
- ジョブ完了と失敗を通知できる
- ジョブ停止ができる

## Open Questions

以下は設計前に詰めたい論点。

- 管理対象は「任意プロセス」か、「特定の起動方法で実行したジョブ」に限定するか
  - SSHubから起動したジョブに限定して良い
- ジョブ開始はアプリから行う前提か、外部で開始したものも監視対象にするか
  - アプリから行う前提で良い
  - 一方で、外部から開始したものも手動で追加できるとなお良い
- 進捗抽出はログ解析ベースで十分か、ジョブ側SDKのような仕組みが必要か
  - ログ解析ベース
- 通知はどの手段を優先するか
  - slackで良い
- 対象環境はデスクトップアプリ、Webアプリ、CLIのどれを主軸にするか
  - GUIがあるものが良い
  - macOS向けデスクトップアプリ前提で良い
- 単一ユーザー向けか、複数ユーザー向けか
  - とりあえずは単一ユーザー向けで良い
- ログの保存期間や保存場所をどうするか
  - localのPCに保存して良い
  - 設計上ssh先に保存したほうが良ければそちらに保存しても良い
  - クラウドは想定していない

## Definition of Success

- 複数SSH先のジョブ状態を一画面で確認できる
- 死んだジョブの原因把握に必要なログへ素早く辿り着ける
- 実行完了や失敗に気づける
- 日常的な学習・simulation監視の手間が減る
